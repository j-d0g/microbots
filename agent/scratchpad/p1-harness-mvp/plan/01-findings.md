## Render Workflows is the execution substrate (not a Web Service per workflow)

Most agents would default to "deploy each user workflow as its own Render Web Service." That's the slow path: 1–5 min Docker build + push + Render API + poll **per user request**. Demo dead air = product killer.

We use **Render Workflows** (public beta) instead. Pre-deploy ONE workflows service hosting a single `run_user_code(code, args)` task. Every user request = SDK call to that task with the generated Python as a string argument. ~12s end-to-end.

**Why this works:** Workflows give Firecracker isolation per run, scale-to-zero billing, and per-second cost. Mapping is direct from an upstream E2B model:
- E2B sandbox per request → Workflow run per request
- E2B template (the bundled deps) → Workflows-service container (we redeploy this only when changing pre-bundled deps, ~1–2 min, rare)

**Gotchas to know up front:**
- **No inbound HTTP.** Workflows are outbound-only. If a demo needs a webhook listener (Slack incoming, etc.), that needs a separate small Web Service relay. P1 demos are chat-driven, so this is fine.
- **4 MB args/run cap.** Large file inputs need out-of-band transfer (S3/R2).
- **Default 2h timeout, max 24h** per task.
- **Cold-start latency is unverified.** Marketing says "milliseconds." Render docs publish no SLO. **Phase 0 must measure this** with a no-op task. If it's >5s consistently, the scratch-task plan needs revision.
- **Pricing:** `starter` $0.05/hr (0.5 CPU, 512 MB), `standard` $0.20/hr (1 CPU, 2 GB), `pro` $0.40/hr (2 CPU, 4 GB). Per-second prorated. Source: https://render.com/docs/workflows-limits
- **Pricing model gap:** Workflows hourly × 730h ≠ Web Service monthly. $0.20/hr × 730 = $146/mo, but Web Service Standard is widely quoted at $25/mo. The two are different cost models — don't mix them in budget math.
- **How to access in the dashboard:** Workflows is **NOT in the left sidebar.** Click `+ New` (top-right) → `Workflow`. Initial creation needs the Render CLI (`brew install render-oss/render/render`) and `render workflows init` to scaffold the project. Source: https://render.com/docs/workflows-tutorial

**Opinionated:** dependencies are pre-bundled in the workflows-service image at deploy time. The scratch-task can't `pip install` mid-run cleanly. Bundle these and don't go bigger (image size affects cold start):

```
composio, httpx, openai, anthropic, pydantic,
beautifulsoup4, pandas, pyyaml, requests
```

If the agent generates code requesting an unlisted lib, surface the error rather than swallowing it.

## Render publishes a Python MCP server template — fork it

`render-examples/mcp-server-python` (https://github.com/render-examples/mcp-server-python). ~70 LOC. FastMCP + Streamable HTTP + `BearerAuthMiddleware` with `hmac.compare_digest`. Bearer token is auto-generated by Blueprint via `MCP_API_TOKEN: { generateValue: true }` — zero-config secret bootstrap.

Tool definition pattern is `@mcp.tool() def fn(name: str) -> str:` — function name, docstring, type hints become the schema. No manual schema declaration.

**Don't write the auth middleware from scratch.** Fork, edit, ship.

## The reference agent loop is much smaller than its monorepo suggests

Numbers an agent in this repo couldn't otherwise know:

- **5 essential MCP tools, not 17 or 24.** The actual `@app.tool()` count differs from upstream documentation. Of those, the load-bearing minimum for an upstream-shaped session is:
  - `consult_docs(paths)` — RAG spine; without it the LLM hallucinates on platform-specific patterns
  - `search_templates(query)` — Build-mode docs mandate template lookup before code-gen
  - `run_code(code, args)` — the workhorse (the scratch-task wrapper)
  - `Ask_User_A_Question(question)` — confirmation gates (system-prompt principles 2 + 4)
  - `Set_Behavior_Mode(mode)` — collapse to a no-op if P1 has only one mode
- `deploy_service` in the upstream stack is **degenerate** — S3 upload + Postgres row + PEP-723 parse. ~50 LOC. No build, no validation, no sandbox spawn. **Don't overengineer this** if/when P1 gets a save-as-workflow path.
- `validate_service` is a *separate* MCP tool, not part of deploy. The upstream agent's prompt encourages calling it before deploy but nothing enforces it. Skippable for P1.
- `launch_sub_agent` does NOT exist on upstream `main` — it's on an unmerged upstream branch. Ignore references to sub-agents in v1 docs.

**One escape-hatch worth knowing for post-P1:** internal sync/async handlers define a `<SERVICE_ID>_SERVICE_URL` env-var proxy path. When set, runtime reverse-proxies to that URL with full HTTP-verb fidelity (sync) or POST-only (async); falls back to sandbox spawn on `ConnectError`. Not used in P1, but if anyone wants to run the upstream stack runtime against a Render-hosted workflow later, this is the hook.

## microbots reuse — verdicts

### Don't lift `microbots/render_sdk/`

Daud's custom code (1,619 LOC, commit `bd889c7`). Real engineering — `requests.Session` over `https://api.render.com/v1`, region aliases, idempotent registry, log streaming.

**Targets the wrong primitive.** It deploys a folder → Render *Web Service* with a stable URL. That's the slow path (1–5 min Docker build + push + Render API + poll **per user request**). Our scratch-task architecture never does this — Workflows are deployed once via git push, then triggered by SDK call.

**Naming-collision warning:** Daud's package is named `render_sdk` (in its `pyproject.toml`). The OFFICIAL Render Workflows SDK on PyPI is also named `render_sdk`. **They are different packages with different APIs.** Don't `pip install` both into the same venv. P1 uses the PyPI one (`from render_sdk import Workflows; @app.task`). Keep the workflows component in its own venv; never run `pip install -e .` from microbots root inside that venv.

Tests in `microbots/render_sdk/tests/` are stubs (3 files, ~4 LOC each). Earlier "5-file pytest suite" claim was overstated.

Daud's SDK is still the right tool for production-faithful "named services per workflow" deploys post-MVP. Not wasted — just not used in P1.

## Architectural decisions

Each decision below is opinionated based on research. The `Why` is the reasoning; treat the `How to apply` as a hard rule unless escalated.

### D1 — Render Workflows + scratch-task for code execution

**Decision:** Pre-deploy ONE Render Workflows service hosting a single `run_user_code(code: str, args: dict | None) -> dict` task. Every agent invocation = `client.workflows.start_task("<workflows-svc>/run_user_code", ...)`.

**Why:** The naive shape (one Render Web Service per saved workflow) creates 1–5 minutes of dead air in the demo every time the agent writes code (Docker build + push + Render API + poll). The scratch-task pattern collapses this to one SDK call (~12s end-to-end). Workflows give us Firecracker isolation per run and scale-to-zero billing. Dependency model is "pre-bundle in image at deploy time."

**How to apply:** Don't add per-workflow Web Services in P1. The single workflows service is the only execution substrate. If a future requirement breaks this (e.g. user wants stable URLs), revisit and log in `notes/decisions-changed.md`.

### D2 — Single-user demo, no auth

**Decision:** Anyone visiting the URL lands in a chat session. No login, no signup. Single shared chat history (or session-cookie-based ephemeral history) for the hackathon.

**Why:** Hackathon scope. Multi-user adds an entire layer (NextAuth, JWT, row-level scoping) that v1 plans for later phases anyway. Time-saving > UX completeness for the demo.

**How to apply:** Don't add login screens. Cookie-based session UUID is sufficient for separating chats during the demo.

### D3 — Minimal Next.js frontend (Vercel AI SDK)

**Decision:** A `useChat`-driven Next.js page. Single route. Vercel AI SDK + `experimental_createMCPClient` connecting to our Render-hosted MCP server with bearer auth.

**Why:** Matches the upstream reference shape — proven pattern. `useChat` + `streamText` is ~100 LOC for the whole UI. Vercel-deployable; or runs via `next start` on a Render Web Service if we want everything on Render.

**How to apply:** Don't reach for a heavier framework or a custom protocol. Streamable HTTP MCP transport is supported out of the box.

### D4 — 5-tool MCP surface for P1

**Decision:** `consult_docs`, `search_templates`, `run_code`, `Ask_User_A_Question`, `Set_Behavior_Mode`. (Reducible to 4 if a single mode is acceptable.)

**Why:** Verified minimum from upstream system prompt + per-mode docs + observed traces. Anything more is gold-plating; anything less breaks the loop.

**How to apply:** Don't add tools without writing the reason in `notes/decisions-changed.md`. `validate_service`, `edit_service`, `deploy_service`, `view_service_code` are all Phase 4+ at earliest.

### D5 — Composio for integrations

**Decision:** When a user workflow needs Slack/Gmail/etc., the generated code uses Composio's Python SDK. The scratch-task container has `composio` pre-installed.

**Why:** Already the team's preference. Composio handles OAuth + tool discovery — saves us writing per-integration glue. No webhooks needed for chat-driven flow; if real-time triggers become important post-P1, add a small Web Service relay.

**How to apply:** Don't write direct Slack SDK / Gmail API calls in P1 templates. Always go through Composio.

### D6 — Postgres for state, no SurrealDB in P1

**Decision:** Render-managed Postgres. Tables: `chat_threads`, `chat_messages`, optionally `templates` (or static JSON file).

**Why:** One DB, simpler ops. v1 plans propose SurrealDB for the memory-graph layer — that's a separate track and not P1's concern. Adopting SurrealDB inherits Docker Compose, schema migrations, and a learning curve for two teammates.

**How to apply:** Don't touch SurrealDB in P1. If a future P-N adds memory, that's when SurrealDB or Desmond's `microbots/knowledge_graph/` gets integrated.

### D7 — Build via Claude Code agent teams

**Decision:** Single long-lived `microbot-harness` team. Opus lead orchestrator + 3 named Sonnet teammates: `infra` (render.yaml + Postgres + scaffolding), `tools` (MCP server + 5 tools), `frontend` (Next.js + chat UI). Phases 0–4 as task-list waves.

**Why:** Per the agent-teams research playbook. Single team minimizes coordination overhead, preserves lead context across phases, and lets us spawn ad-hoc verifiers when needed. Multiple teams burn coordination tokens for no gain at this scale.

**How to apply:** One team only. If a teammate is blocked, spawn a one-off verifier agent in the same team to drill into the blocker — don't fork a new team.
