## Render Workflows is the execution substrate (not a Web Service per workflow)

Most agents would default to "deploy each user workflow as its own Render Web Service." That's the slow path: 1–5 min Docker build + push + Render API + poll **per user request**. Demo dead air = product killer.

We use **Render Workflows** (public beta) instead. Pre-deploy ONE workflows service hosting a single `run_user_code(code, args)` task. Every user request = SDK call to that task with the generated Python as a string argument. ~12s end-to-end.

**Why this works:** Workflows give Firecracker isolation per run, scale-to-zero billing, and per-second cost. Mapping is direct from an upstream E2B model:
- E2B sandbox per request → Workflow run per request
- E2B template (the bundled deps) → Workflows-service container (we redeploy this only when changing pre-bundled deps, ~1–2 min, rare)

**Gotchas to know up front:**
- **No inbound HTTP.** Workflows are outbound-only. If a demo needs a webhook listener (Slack incoming, etc.), that needs a separate small Web Service relay. P1 demos are chat-driven, so this is fine.
- **4 MB args/run cap.** Large file inputs need out-of-band transfer (S3/R2).
- **Default 2h timeout, max 24h** per task.
- **Cold-start latency is unverified.** Marketing says "milliseconds." Render docs publish no SLO. **Phase 0 must measure this** with a no-op task. If it's >5s consistently, the scratch-task plan needs revision.
- **Pricing:** `starter` $0.05/hr (0.5 CPU, 512 MB), `standard` $0.20/hr (1 CPU, 2 GB), `pro` $0.40/hr (2 CPU, 4 GB). Per-second prorated. Source: https://render.com/docs/workflows-limits
- **Pricing model gap:** Workflows hourly × 730h ≠ Web Service monthly. $0.20/hr × 730 = $146/mo, but Web Service Standard is widely quoted at $25/mo. The two are different cost models — don't mix them in budget math.

**Opinionated:** dependencies are pre-bundled in the workflows-service image at deploy time. The scratch-task can't `pip install` mid-run cleanly. Bundle these and don't go bigger (image size affects cold start):

```
composio, httpx, openai, anthropic, pydantic,
beautifulsoup4, pandas, pyyaml, requests
```

If the agent generates code requesting an unlisted lib, surface the error rather than swallowing it.

## Render publishes a Python MCP server template — fork it

`render-examples/mcp-server-python` (https://github.com/render-examples/mcp-server-python). ~70 LOC. FastMCP + Streamable HTTP + `BearerAuthMiddleware` with `hmac.compare_digest`. Bearer token is auto-generated by Blueprint via `MCP_API_TOKEN: { generateValue: true }` — zero-config secret bootstrap.

**Don't write the auth middleware from scratch.** Fork, edit, ship.

## The reference agent loop is much smaller than its monorepo suggests

Numbers an agent in this repo couldn't otherwise know:

- **5 essential MCP tools, not 17 or 24.** The actual `@app.tool()` count differs. Of those, the load-bearing minimum for an upstream agent-shaped session is:
  - `consult_docs(paths)` — RAG spine; without it the LLM hallucinates on platform-specific patterns
  - `search_templates(query)` — Build-mode docs mandate template lookup before code-gen
  - `run_code(code, args)` — the workhorse (the scratch-task wrapper)
  - `Ask_User_A_Question(question)` — confirmation gates (system-prompt principles 2 + 4)
  - `Set_Behavior_Mode(mode)` — collapse to a no-op if P1 has only one mode
- `deploy_service` in the upstream stack is **degenerate** — S3 upload + Postgres row + PEP-723 parse. ~50 LOC. No build, no validation, no sandbox spawn. **Don't overengineer this** if/when P1 gets a save-as-workflow path.
`validate_service` is a *separate* MCP tool, not part of deploy. The upstream agent's prompt encourages calling it before deploy but nothing enforces it. Skippable for P1.
- `launch_sub_agent` does NOT exist on the upstream codebase `main` — it's on an unmerged upstream branch. Ignore references to sub-agents in v1 docs.

**One escape-hatch worth knowing for post-P1:** internal sync/async handlers define a `<SERVICE_ID>_SERVICE_URL` env-var proxy path. When set, runtime reverse-proxies to that URL with full HTTP-verb fidelity (sync) or POST-only (async); falls back to sandbox spawn on `ConnectError`. Not used in P1, but if anyone wants to run the upstream stack runtime against a Render-hosted workflow later, this is the hook.

## microbots reuse — verdicts

### Don't lift `microbots/render_sdk/`

Daud's custom code (1,619 LOC, commit `bd889c7`). Real engineering — `requests.Session` over `https://api.render.com/v1`, region aliases, idempotent registry, log streaming.

**Targets the wrong primitive.** It deploys a folder → Render *Web Service*. Our architecture uses Render *Workflows*, which has a totally different deploy model (git push, not API-driven Docker builds). Lifting it adds 1,619 LOC of unused infrastructure.

**Naming-collision warning:** Daud's package is named `render_sdk` (in its `pyproject.toml`). The OFFICIAL Render Workflows SDK on PyPI is also named `render_sdk`. **They are different packages with different APIs.** Don't `pip install` both into the same venv. P1 uses the PyPI one (`from render_sdk import Workflows; @app.task`).

Tests in `microbots/render_sdk/tests/` are stubs (3 files, ~4 LOC each). The earlier "5-file pytest suite" claim was overstated.

### Optionally lift `microbots/log.py`

~280 LOC Logfire facade by Daud. Single file. `get_logger`, `span`, `instrument`, `get_correlation_id`. Worth ~30 min if observability is wanted.

### Don't extend `microbots/knowledge_graph/`

Desmond's product (10,428 LOC) — SurrealDB-backed user-behavior memory graph + Composio ingest + Pydantic AI wiki agent. **No MCP server, no agent loop, no frontend.** Different product from P1.

The Pydantic AI tool-registration pattern at `wiki/agent.py:74-93` and the named-query pattern at `db/queries.py` are useful as *references*. Don't import the code.

`agent/PLAN-v1.md` self-flags as stale ("Implementation plan written before the team locked the actual hackathon schedule"). Don't follow.

## Composio for integrations

Team preference (per `agent/BRAINDUMP.md` Martin transcript). Composio handles OAuth + tool discovery — no per-integration glue needed.

- Pre-install `composio` in scratch container.
- Composio is the integration layer; user code calls `composio.actions.execute(...)` directly.
- For chat-driven flows, no inbound webhooks needed — outbound calls only.
- For real-time triggers (Slack message → workflow run), eventually need a small Render Web Service relay. Out of P1.
